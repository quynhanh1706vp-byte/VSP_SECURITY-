## 4. Root cause

### Trigger
TBD — awaiting investigation. Candidates:
- Upstream payment gateway API change
- Database schema migration side effect (Sprint 4 work)
- Webhook delivery queue congestion
- Auth token rotation breaking background billing jobs

### Contributing factors (process)

1. **No verification step before closing incident** — SD-0049 was marked resolved
   without a "health check window" proving symptoms didn't return.

2. **No postmortem requirement** — closure didn't require documenting what was fixed,
   which created no forcing function to verify the fix actually worked.

3. **No automated re-detection** — SIEM correlation rules didn't alert when billing
   errors resurged after the initial "fix", so the regression went unnoticed for an
   unknown amount of time.

### Detection gap

The gap between first close and reopen at `bb53769` represents silent time where the
outage was happening but nobody knew. We don't currently have:
- Heartbeat monitoring on billing webhook delivery
- Error rate SLO with alerting
- Post-incident watch period automation

- Gap between closes was not monitored
- No communication to affected users during either outage period

## 9. Lessons learned

1. **"Resolved" ≠ "Closed"**. Resolved means symptoms stopped. Closed requires postmortem
   + verification window + monitoring confirms no recurrence. These should be two
   distinct states with different exit criteria.

2. **Monitoring must outlive the incident**. After declaring resolution, monitoring
   should watch for regression for at least 24h. Silent regression is the worst case.

3. **Postmortem discipline prevents premature close**. If every close requires writing
   "what went wrong and how we fixed it", premature closures become obviously thin
   during review.
